# F7.5-01-PLAN ŌĆö Open-Meteo connector + silver refactor (single plan)

## Goal

Land the full Workstream A + B + D refactor in this repo so silver carries
hub-height wind, irradiance components, cloud decomposition, snow, and
derived air density across three role-specific datasets (demand, wind,
solar) at three role-specific location lists. The existing demand path
(`historical` / `forecast`) is renamed (not aliased) to
`historical_demand` / `forecast_demand` and remains green via updated
unit tests. Workstream C (15-min) and the wind/solar bronze backfill are
out of scope and live in backlog.

Goal-backward: a downstream consumer running

```python
from gridflow.silver.openmeteo import HistoricalWindWeather
HistoricalWindWeather(...).run(target_date=date(2025, 1, 15), run_id=...)
```

produces a silver Parquet at `data/silver/open_meteo/historical_wind/...`
with columns including `wind_speed_100m`, `wind_direction_100m`,
`wind_gusts_10m`, `cloud_cover`, `cloud_cover_low/mid/high`, `dew_point_2m`,
`air_density_kg_m3`, plus the F0 bitemporal columns (`event_time`,
`available_at`, `source_run_id`, `dataset_version="2.0.0"`). The wind silver
covers all 12 capacity-weighted GB sites listed in F7.5-LOC-01.

## Sequencing rationale

**Three commits, sequential.** A pre-execution review found that wave 1 in
isolation cannot stay green ŌĆö `tests/unit/test_openmeteo.py` and
`tests/endpoints/test_endpoint_urls.py` import `WeatherObservation`,
`OpenMeteoConnector`, `LOCATIONS`, and `HOURLY_VARIABLES` directly and
exercise the connector with `dataset="historical"`. After the spec-table
refactor, that key no longer exists. Combine the refactor + schemas +
transformers + tests into a single atomic commit (consistent with the
user's "hard rename + update callers in same commit" decision).

## Pre-execution facts (verified at plan time)

- **`BaseSchema`** in `src/gridflow/schemas/common.py` is bare:
  `model_config = {"strict": True, "extra": "ignore"}`. It does NOT
  declare bitemporal columns. The F0 pattern stamps `event_time`,
  `available_at`, `source_run_id`, `dataset_version` on the Polars
  DataFrame at silver-write time, NOT on the Pydantic schema. The three
  new schemas therefore do NOT declare bitemporal fields either.
- **Air density at the joint extremes:** running the formula at the
  documented limits gives:
  - `T = -30┬░C, P = 950 hPa` ŌåÆ Žü = 1.361 kg/m┬│
  - `T = -30┬░C, P = 1050 hPa` ŌåÆ Žü = 1.504 kg/m┬│ (above 1.40)
  - `T = +45┬░C, P = 950 hPa` ŌåÆ Žü = 1.040 kg/m┬│
  - `T = +45┬░C, P = 1050 hPa` ŌåÆ Žü = 1.150 kg/m┬│

  The property-test band is therefore `[0.95, 1.55]`, not `[0.95, 1.40]`.
  The CONTEXT requirement F7.5-VARS-06 referenced the narrower band; the
  plan-time verification widens it.

- **`tests/endpoints/test_endpoint_urls.py:578-583`** hardcodes the
  expected `HOURLY_VARIABLES` list. After the rename to
  `DEMAND_HOURLY_VARS`, that hardcoded list MUST gain `"snowfall"` and
  `"snow_depth"` entries to match the new demand variable set
  (per F7.5-VARS-05).

- **`tests/unit/test_openmeteo.py:177-216`** exercises the reingest path
  using bronze dataset names `forecast_london` and `historical_london`.
  Both must be renamed to the new double-underscore forms
  `forecast_demand__london` and `historical_demand__london`.

- **`tests/unit/test_openmeteo.py:226-272`** validates `WeatherObservation`.
  After the schema split, this class either becomes
  `TestDemandWeatherSchema` (against `DemandWeather`) or is split into
  three test classes (one per schema). Recommendation: one test class per
  schema, three classes total ŌĆö the existing four assertions get copied
  per schema with field set adjusted.

### Commit 3 ŌĆö vault docs + ADR

   - **Context:** F7.5 introduces 18 capacity-weighted GB sites for wind and solar weather sampling. The exact lat/lon are approximate centroids drawn from public capacity registers (TEC register, DUKES), not vendor coordinates from operator NRO submissions.
