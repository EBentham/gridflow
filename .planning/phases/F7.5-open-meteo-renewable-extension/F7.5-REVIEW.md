# Phase F7.5: Code Review Report

**Reviewed:** 2026-05-09
**Depth:** deep
**Files Reviewed:** 15
**Status:** issues_found

## Summary

The role-split refactor of the Open-Meteo connector is internally consistent: `endpoints.py`, `client.py`, the three new schemas, the six transformer subclasses, the YAML config, and the rewritten test suite all hang together correctly. The dataclass-with-`default_factory=tuple` pattern is correct (the `tuple` type is callable and yields `()`); the `WeatherObservation` / `LOCATIONS` / `HOURLY_VARIABLES` deletions have no remaining importers in `src/`, `scripts/`, or `tests/`; the double-underscore bronze identifier matches between writer (`client.py:112`) and reader (`historical.py:104`); and `_pivot_openmeteo_json`'s new `hourly_vars` parameter is passed correctly by every caller.

The refactor leaks at the **integration boundaries** the planning never re-walked. Two stragglers reference the deleted `historical` / `forecast` dataset names by string literal:

1. **`src/gridflow/serving/client.py:134`** — `GridflowClient.get_weather()` selects from `silver_historical`. After F7.5 the only auto-registered openmeteo views are `silver_historical_demand`, `silver_historical_wind`, `silver_historical_solar` and the three forecast variants. `silver_historical` no longer exists, so calling `get_weather()` against a properly-built catalogue raises a DuckDB `Catalog Error`. `GridflowClient` is the publicly-documented Python SDK (per `docs/PROJECT_OVERVIEW.md`) — F7.5 silently broke a SDK method.

2. **`scripts/run_all_sources.py:70`** — the public-source map still lists `"open_meteo": ["historical", "forecast"]`. The connector's `fetch()` (`client.py:61-65`) now `raise ValueError`s on unknown dataset names, so this script logs `FAILED: Unknown open_meteo dataset 'historical'; expected one of [...]` for both datasets when run. The wider script swallows the error per-dataset; nightly local runs silently regress openmeteo ingestion to nothing.

CR-01 and WR-01 are the **same class of bug**: F7.5 deleted the dataset names `historical` and `forecast`, and downstream code referenced those names by string literal in places the refactor's grep sweep didn't reach. Before merging the user should grep the whole repository for `'historical'`, `"historical"`, `'forecast'`, `"forecast"` (in source and config — I swept `src/`, `scripts/`, and `tests/`, but not `docs/`, `config/` flavour-files, notebooks, or any external `quant-vault/` references) to confirm no other call sites.

Plus a hacky test artefact (`assert ... if False else True` is always `True`), three stale doc-string lines in `scripts/`, an unused `SCHEMA` ClassVar on the transformers, and a comment in `silver/base.py:198` that still references the deleted `historical_london` partition. The CLAUDE.md hard rules (no pandas, no f-string SQL on user data, polars only, `os.replace()` atomic writes, `from __future__ import annotations`) are all satisfied within F7.5's diff.

## Critical Issues

### CR-01: GridflowClient.get_weather() queries deleted view `silver_historical`

**File:** `src/gridflow/serving/client.py:131-137`
**Severity:** HIGH

**Issue:** `GridflowClient.get_weather()` issues `SELECT … FROM silver_historical`. F7.5 splits the openmeteo silver dirs into `historical_demand`/`historical_wind`/`historical_solar`, and `_register_views()` in `storage/duckdb.py:120` auto-creates view names from `dataset_dir.name`, so the registered views are now `silver_historical_demand` etc. — `silver_historical` is never created. Any consumer calling `get_weather()` against a freshly-rebuilt catalogue gets a DuckDB catalog error.

```python
return self.query(f"""
    SELECT timestamp_utc, location, latitude, longitude,
           temperature_2m, wind_speed_10m, precipitation, hdd, cdd
    FROM silver_historical                         #  <-- view does not exist
    WHERE {where}
    ORDER BY timestamp_utc, location
""")
```

`GridflowClient` is the public Python SDK (per `docs/PROJECT_OVERVIEW.md:100,231` and `.planning/codebase/ARCHITECTURE.md:136,256`); breakage here is a public-API regression. There are no in-tree callers other than the SDK module itself, so the test suite did not catch it. Note `.planning/codebase/CONCERNS.md:45-49` already flagged a sibling case (`GridflowClient.get_generation_by_fuel()` queries a view that doesn't exist either), so this is a recurring class of un-tested SDK drift.

**Fix:** Point `get_weather()` at `silver_historical_demand` (the closest preserved equivalent to F0's `silver_historical` for the demand population centres):

```python
def get_weather(
    self,
    start: str | date,
    end: str | date,
    location: str | None = None,
) -> pl.DataFrame:
    where_clauses = [f"timestamp_utc::DATE BETWEEN '{start}' AND '{end}'"]
    if location:
        where_clauses.append(f"location = '{location}'")
    where = " AND ".join(where_clauses)
    return self.query(f"""
        SELECT timestamp_utc, location, latitude, longitude,
               temperature_2m, wind_speed_10m, precipitation, hdd, cdd
        FROM silver_historical_demand
        WHERE {where}
        ORDER BY timestamp_utc, location
    """)
```

A follow-up commit could add `get_wind_weather()` / `get_solar_weather()` (or a `role` parameter) if downstream consumers want the new datasets exposed via the SDK. The f-string SQL injection pattern across this module is pre-existing (already documented in `.planning/codebase/CONCERNS.md:10`) and out of F7.5 scope.

## Warnings

### WR-01: scripts/run_all_sources.py iterates deleted dataset names

**File:** `scripts/run_all_sources.py:70`
**Severity:** MEDIUM

**Issue:** The public-source dataset map still lists the pre-F7.5 names:

```python
"open_meteo": ["historical", "forecast"],
```

When the script is executed (the docstring shows it is intended as the canonical "run everything for one date" debug entrypoint), `connector.fetch("historical", ...)` reaches `OpenMeteoConnector.fetch` (`client.py:61-65`), which `raise ValueError`s on unknown names. The wider script catches the exception per-dataset and logs `FAILED: Unknown open_meteo dataset 'historical'; expected one of [...]`, so the run does not crash but produces zero openmeteo bronze for the day. Every nightly local run with this script silently regresses openmeteo ingestion to nothing.

**Fix:** Replace the pair with the six F7.5 dataset names:

```python
"open_meteo": [
    "historical_demand", "historical_wind", "historical_solar",
    "forecast_demand", "forecast_wind", "forecast_solar",
],
```

(Out-of-scope but worth a note: lines 53-54 still list `bod` (decommissioned per `config/sources.yaml:18`) and line 67 lists `generation_by_fuel` (removed per `config/sources.yaml:79`). Pre-F7.5 stragglers, but the same sweep that fixes the openmeteo pair could fix these too.)

### WR-02: Hacky test line evaluates to a no-op assertion

**File:** `tests/unit/test_openmeteo_air_density.py:107`
**Severity:** MEDIUM

**Issue:** The line:

```python
assert "air_density_kg_m3" not in HistoricalSolarWeather()._output_columns() if False else True  # noqa: E501
```

is parsed by Python as `assert ((...not in...) if False else True)` — the conditional collapses to the `else` branch, the assertion always sees `True`, and the actual `_output_columns()` check is **never executed**. This silently disables the contract this test purports to enforce. The `# noqa: E501` masks a long-line warning the original author probably needed because the line was being rewritten when the task was paused.

Independently of the dead assertion: `HistoricalSolarWeather()` (the bare-`()` constructor call) would have raised `TypeError: __init__() missing 1 required positional argument: 'data_dir'` from `BaseSilverTransformer.__init__`. So this line is dead code that was never actually testable in this form — the test's goal (`"air_density_kg_m3" not in solar output cols`) is salvaged by the next block (line 109-134), which constructs the transformer with `__new__` and asserts the column absence on the produced DataFrame.

**Fix:** Delete line 107 entirely. The assertion on the same condition exists at line 134 (`assert "air_density_kg_m3" not in out.columns`) and is real.

### WR-03: `BaseOpenMeteoTransformer.SCHEMA` is declared but never used

**File:** `src/gridflow/silver/openmeteo/historical.py:90, 215, 237, 256`
**Severity:** MEDIUM

**Issue:** `BaseOpenMeteoTransformer` declares `SCHEMA: ClassVar[type[BaseSchema]]` and each subclass binds it to `DemandWeather`/`WindWeather`/`SolarWeather`, but `transform()` never instantiates the schema or runs any per-row validation against it. Other vendors (`silver/entsoe/h6_market.py:115`, `silver/entsoe/outages_h7.py:109`) call `self.schema_cls(**df.row(0, named=True))` to enforce the contract; openmeteo does not. The schemas exist as documentation only — invalid types or naive timestamps in the bronze JSON will pass straight through to silver Parquet as long as Polars' `cast(pl.Float64)` accepts them. The new `_BaseWeather.must_be_utc` validator (added with the F7.5 split) is unreachable for the same reason.

This is the same gap as F0 (the pre-existing `HistoricalWeatherTransformer` did not validate either), so F7.5 only inherits it. But adding `SCHEMA` as a `ClassVar` plus a `field_validator` on the schema implies validation is happening when it isn't — that mismatch is worth resolving.

**Fix:** Either (a) wire the schema into `transform()` — e.g. validate the first row in `_add_derived` to fail loudly on schema drift, mirroring the entsoe transformers — or (b) drop the `SCHEMA` ClassVar and the `_BaseWeather` validator and document explicitly that openmeteo silver relies on Polars cast for type coercion. (a) is the safer choice now that the schemas are exported as public API in `silver/openmeteo/__init__.py`.

## Info

### IN-01: Module-level `assert` rendered no-op under `python -O`

**File:** `src/gridflow/silver/openmeteo/historical.py:264-266` and `src/gridflow/silver/openmeteo/forecast.py:70-72`
**Severity:** LOW

**Issue:** Both files contain blocks like:

```python
assert HistoricalDemandWeather.HOURLY_VARS == DATASET_SPECS["historical_demand"].hourly
```

`assert` statements are stripped by the bytecode compiler when Python is run with `-O`. The current project does not deploy with `-O` (it is local-first per CLAUDE.md and tests run without it), so this is a defensive-coding nit, not a live defect. If a future packaged deployment ever adds `-O` (or some IDE runs the module through a `-O`-enabled interpreter), the import-time drift check goes silent.

**Fix:** Optional — replace with a real `if … : raise RuntimeError(...)` if you want the check to survive `-O`. Otherwise leave as is; the contract test at `tests/unit/test_openmeteo_dataset_specs.py:test_six_dataset_keys_present` covers the location wiring (though not the per-transformer `HOURLY_VARS` binding, which is an adjacent gap to consider).

### IN-05: `is_historical = dataset.startswith("historical")` is brittle

**File:** `src/gridflow/connectors/openmeteo/client.py:90`
**Severity:** LOW

**Issue:** `_fetch_location` decides which base URL to hit by string-prefix matching the dataset name. This couples the URL choice to the lexical key spelling. Any future role added with a non-prefixed key (e.g. `era5_demand`) would need a separate change here. There is no test that the prefix matches an enumerated set; an entry like `historicized_wind` would silently take the archive branch.

**Fix:** Drive the URL choice from a third field on `WeatherDatasetSpec` (e.g. `archive: bool`), or partition `DATASET_SPECS` into `ARCHIVE_DATASETS` / `FORECAST_DATASETS` constants.

### IN-06: `read_bronze` swallows AttributeError silently

**File:** `src/gridflow/silver/openmeteo/historical.py:120-123`
**Severity:** LOW

**Issue:** The `try ... except (json.JSONDecodeError, AttributeError, KeyError)` block in `read_bronze` catches `AttributeError` and `KeyError` then logs a warning. `AttributeError` would only fire if `data` is unexpectedly not a dict — but at that point the bronze write itself is corrupted and we would prefer to fail loudly rather than silently dropping the location/day. The CLAUDE.md project rule "Pydantic validation failures are logged and surfaced — never silently dropped" applies by analogy.

**Fix:** Narrow to `except json.JSONDecodeError` and let `KeyError`/`AttributeError` propagate. Or use `logger.exception(...)` so the stack trace lands in the log for diagnosis.

### IN-07: `_BaseWeather.location: str` accepts any string, not enumerated

**File:** `src/gridflow/schemas/weather.py:27`
**Severity:** LOW

**Issue:** The schema field `location: str` is unconstrained. A typo in `BRONZE_DATASET_PREFIX` or a manual reingest of the wrong file would produce silver rows with `location="hornesa"` (sic) and pass validation. The 7+12+6 = 25 location names across the three lists are a closed set; a `Literal[...]` union or a runtime `field_validator` would catch the mistake at write time.

**Fix:** Optional cleanup — would tighten the contract; not required for F7.5 to ship.
