dataset families (demand, wind, solar) with role-specific location lists
and variable sets.

## Re-ingest commands (DOCUMENTED, NOT EXECUTED)

The new `historical_wind` and `historical_solar` datasets have no
historical data on disk. A backfill is order:

- 12 wind sites × 8 years × 365 days ≈ 35,000 location-days
- 6 solar sites × 8 years × 365 days ≈ 17,500 location-days
- Total ≈ 52,500 API calls @ 5 req/sec ≈ 175 minutes wall-clock

Per CLAUDE.md "No live API ingestion against a real key without explicit
user confirmation", these commands are not run during phase execution.
Recorded here for the user (or a subsequent live-execute session):

```bash
# Wind backfill
gridflow ingest open_meteo historical_wind --start 2018-01-01 --end 2025-12-31
gridflow transform open_meteo historical_wind --start 2018-01-01 --end 2025-12-31

# Solar backfill
gridflow ingest open_meteo historical_solar --start 2018-01-01 --end 2025-12-31
gridflow transform open_meteo historical_solar --start 2018-01-01 --end 2025-12-31

# Demand re-transform (to pick up the new snowfall / snow_depth /
# air_density_kg_m3 columns; existing bronze is reused).
# Bronze sidecars are required for correct historical `available_at`.
gridflow transform open_meteo historical_demand \
    --start 2024-01-01 --end 2025-12-31 --reingest
gridflow transform open_meteo forecast_demand \
    --start 2024-01-01 --end 2025-12-31 --reingest
```

The Avast `curl --ssl-no-revoke` workaround locked in V1-CONTEXT.md
applies to any verification curl. Backfill itself runs through Python
httpx via the connector — Avast TLS interception broke the local
`uv pip install` step in this session (PYTHONPATH=src was used
throughout for tests), but live API calls from the connector are not
blocked because Open-Meteo's certificate chain is one Avast already
inspects without rejection. If a future live run sees `UnknownIssuer`,
fall back to the V1 curl-based pattern.

## What comes next

When `gridflow_models` (the external project referenced throughout the
proposed plan) is ready to consume the new silver, its wind and solar
feature pipelines should:

1. Read from `data/silver/open_meteo/historical_wind/` and
   `data/silver/open_meteo/historical_solar/` rather than the F0-era
   `historical/` path.
2. Use `wind_speed_100m_ms` (or whatever name the F6 codebase chooses)
   from the new silver schema, not `wind_speed_10m_ms` from the
   F0-era schema. Wind shear features can be derived as the implied
   `wind_speed_100m / wind_speed_10m` ratio.
3. Consume the GHI/DNI/DHI/GTI components for solar and the cloud-cover
   decomposition for both wind and solar feature pipelines.

The `cloud_cover_pct` field name referenced by the proposed plan does
not exist; `WindWeather` carries `cloud_cover` (total, %), plus the
three altitude bands. F6's wind feature pipeline needs to adopt the new
column name when it eventually integrates.
