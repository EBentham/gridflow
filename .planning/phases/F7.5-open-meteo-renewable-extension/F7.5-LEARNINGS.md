# Phase F7.5 Learnings: Open-Meteo Connector Extension for Renewable Forecasting

## Decisions

### Hard rename with no compatibility aliases

The old dataset names `historical` and `forecast` were deleted outright and replaced with role-specific names (`historical_demand`, `forecast_demand`, `historical_wind`, `historical_solar`, `forecast_wind`, `forecast_solar`). No transition aliases, no `WeatherObservation` shim.

**Rationale:** CLAUDE.md project rule: "No backwards-compatibility hacks like renaming unused _vars, re-exporting types, adding // removed comments for removed code." Aliases would have delayed the same breakage without reducing it, and callers are in a single local repo with a complete test suite to surface the change.
**Source:** F7.5-01-PLAN.md (Sequencing rationale)

---

### Wave 1 + Wave 2 + Tests collapsed into single atomic commit

The original plan had a wave-1-alone commit (endpoints refactor only). Pre-execution advisor review found that `tests/unit/test_openmeteo.py` and `tests/endpoints/test_endpoint_urls.py` import `WeatherObservation`, `LOCATIONS`, and `HOURLY_VARIABLES` directly, so after the endpoints refactor alone the test suite is red. The plan was revised: connector + schemas + transformers + all test updates land in one atomic commit.

**Rationale:** "A pre-execution review found that wave 1 in isolation cannot stay green." Consistent with the rule that tests must pass at every commit boundary.
**Source:** F7.5-01-PLAN.md (Sequencing rationale)

---

### Archive hub-heights restricted to 10m + 100m; forecast carries wider set

`WIND_ARCHIVE_VARS` deliberately excludes `wind_speed_80m`, `wind_speed_120m`, `wind_speed_180m`. `WIND_FORECAST_VARS` includes the full hub-height set.

**Rationale:** Live ERA5 archive probe on 2026-05-09 confirmed 80/120/180m return `units: "undefined"` and all-null values at both Hornsea and Whitelee. Using those variables for archive requests would silently produce null-filled silver. Forecast endpoint carries the wider set from GFS/ICON which interpolates the missing heights.
**Source:** F7.5-VERIFICATION.md (Observable Truth #3); F7.5-01-PLAN.md (Pre-execution facts)

---

### Double-underscore separator for per-location bronze identifiers

Bronze parquet files are named `{dataset}__{location.name}` (e.g. `historical_demand__london`). Single underscore would be ambiguous when the dataset prefix is itself multi-word.

**Rationale:** `historical_wind_hornsea` could be parsed as dataset=`historical`, subdataset=`wind`, location=`hornsea` OR dataset=`historical_wind`, location=`hornsea`. The double underscore is an unambiguous delimiter. Contract test enforces that no location name itself contains `__`.
**Source:** F7.5-01-PLAN.md (Task breakdown); F7.5-VERIFICATION.md (Observable Truth #5)

---

### Air density band widened to [0.95, 1.55] kg/m³

Property test bounds widened from the original CONTEXT spec of `[0.95, 1.40]` to `[0.95, 1.55]`.

**Rationale:** Plan-time formula evaluation at the joint extreme T=-30°C, P=1050 hPa gives ρ ≈ 1.504 kg/m³, which exceeds 1.40. The original bound was derived from typical UK winter conditions without stress-testing the cold+high-pressure joint extreme. Arctic air masses can bring these conditions.
**Source:** F7.5-01-PLAN.md (Pre-execution facts — air density at joint extremes)

---

### Location centroids, not per-turbine NRO coordinates (ADR-020)

The 12 wind and 6 solar site coordinates are approximate capacity-weighted centroids from public registers (TEC register, DUKES), not the precise vendor coordinates from operator NRO submissions.

**Rationale:** Per-turbine NRO coordinates are not freely available. Spatial precision is acceptable for boostable feature inputs at current modelling resolution. A follow-up can replace if residual analysis surfaces coordinate sensitivity.
**Source:** docs/DECISION_LOG/ADR-020-openmeteo-location-approximation.md

---

### GTI solar parameters injected via frozen extra_params tuple on WeatherDatasetSpec

Solar datasets carry `extra_params=(("tilt","35"),("azimuth","180"))` on their `WeatherDatasetSpec`. The client materialises these into request params via `request_params.update(dict(spec.extra_params))`.

**Rationale:** Avoids a separate code path for solar; the spec dataclass is the single source of truth for all per-dataset variations (locations, variables, request parameters). A contract test enforces the invariant: `extra_params` is non-empty if and only if `global_tilted_irradiance` is in `spec.hourly`.
**Source:** F7.5-01-PLAN.md (Task breakdown §2); F7.5-VERIFICATION.md (Threat #5)

---

### DATASET_VERSION bumped 1.0.0 → 2.0.0 to signal schema incompatibility

All six transformer subclasses declare `DATASET_VERSION = "2.0.0"`. The demand re-transform recommendation uses `--reingest` to reprocess existing bronze under the new schema.

**Rationale:** The new schemas (`DemandWeather`, `WindWeather`, `SolarWeather`) differ in field set from F0's `WeatherObservation`. Silver Parquet from F0 has different columns. A version bump ensures consumers can detect schema drift and reject stale silver rather than reading mismatched columns silently.
**Source:** F7.5-01-PLAN.md (Task breakdown); F7.5-RESULTS.md (Re-ingest commands)

## Lessons

### Migration grep sweep must cover the whole repo, not just the modified tree

F7.5's migration sweep grepped `src/gridflow/connectors/openmeteo/`, `silver/openmeteo/`, and `cli.py` for old dataset names. It missed `src/gridflow/serving/client.py` (which queried `FROM silver_historical` by string literal) and `scripts/run_all_sources.py` (which iterated `["historical", "forecast"]`). Both were caught by the post-execution code review — not by tests.

**Context:** Code review (CR-01 + WR-01) classified these as the same bug class: "F7.5 deleted the dataset names `historical` and `forecast`, and downstream code referenced those names by string literal in places the refactor's grep sweep didn't reach." String-literal references to API/dataset names in scripts and serving code are invisible to import-graph analysis — only a repo-wide string grep catches them. After any hard rename, the sweep should be: `grep -r "'historical'" src/ scripts/ tests/` and `grep -r '"historical"' src/ scripts/ tests/`.
**Source:** F7.5-REVIEW.md (Summary + CR-01 + WR-01)

---

### Always calculate formula bounds at joint extremes before writing property tests

The CONTEXT document specified an air density band of `[0.95, 1.40]`. The plan-time formula evaluation found T=-30°C combined with P=1050 hPa yields 1.504 kg/m³ — exceeding the documented bound. The bound was widened before writing the tests; if the tests had been written first using the narrower bound they would have been wrong from day one.

**Context:** The original 1.40 upper bound was based on typical UK conditions. Arctic outbreaks combine cold temperatures with blocking highs, reaching the joint extreme. Any physical formula with two independent parameters needs corner-case evaluation at all four (min/max) × (min/max) combinations, not just at typical values.
**Source:** F7.5-01-PLAN.md (Pre-execution facts — air density)

---

### Transitive test imports must be audited before planning multi-wave refactors

Wave 1 in isolation (endpoints refactor only) appeared self-contained from a source-code perspective. But `tests/unit/test_openmeteo.py` imported `WeatherObservation`, `LOCATIONS`, and `HOURLY_VARIABLES` by name — all deleted by wave 1. This was caught by a pre-execution advisor review and avoided a broken intermediate commit, but only because the advisor checked test imports explicitly.

**Context:** A wave-based refactor plan is only valid if each wave boundary leaves the test suite green. Before finalising sequencing, grep all test files for every symbol being deleted or renamed. If any test imports a soon-deleted name, that test must move to the same wave as the deletion.
**Source:** F7.5-01-PLAN.md (Sequencing rationale)

---

### BaseSchema does NOT declare bitemporal columns — they are stamped at silver-write time

`BaseSchema` in `src/gridflow/schemas/common.py` is bare (`model_config` only). The F0 bitemporal columns (`event_time`, `available_at`, `source_run_id`, `dataset_version`) are stamped on the Polars DataFrame by `BaseSilverTransformer._add_bitemporal_columns()` at write time, not declared on Pydantic schemas. Adding them to the new schemas would have been wrong.

**Context:** This is a non-obvious split: the schema is for validating a single record as typed by the API, while bitemporal columns are pipeline metadata added after validation. Other Pydantic-first projects might declare them on the model, but gridflow's pattern is: schema = API record shape only.
**Source:** F7.5-01-PLAN.md (Pre-execution facts — BaseSchema); F7.5-VERIFICATION.md (Observable Truth #9)

### Vault is symlinked content and cannot be updated without obsidian-vault MCP access

`quant-vault/30-vendors/open-meteo/` is an Obsidian vault directory listed in `.gitignore` and not present in the worktree. F7.5-VAULT-01 (update vault documentation) could only be marked partial because the vault requires a separate MCP server session to write.

**Context:** Any phase that includes a "vault docs" requirement should note upfront that this step requires the `obsidian-vault` MCP server to be connected. Without it, the in-repo `docs/` equivalents are the version-controlled substitute, but the vault paper-trail remains stale. Verify MCP server availability at phase-start, not at vault-step time.
**Source:** F7.5-RESULTS.md (Vault note); F7.5-VERIFICATION.md (Deferred: F7.5-VAULT-01)

## Patterns

### WeatherDatasetSpec frozen dataclass as single source of truth

All per-dataset variation (locations, hourly variable list, extra request parameters) is captured in a frozen `WeatherDatasetSpec` dataclass keyed in `DATASET_SPECS: dict[str, WeatherDatasetSpec]`. The connector, transformer, config loader, and tests all derive from this single table.

**When to use:** When a single connector serves N role-differentiated datasets that share the same fetch/transform logic but differ in location lists, variable sets, and optional request parameters. Centralising variation in a spec table eliminates drift: adding a new dataset is one `DATASET_SPECS` entry plus a transformer subclass.
**Source:** F7.5-01-PLAN.md (Task breakdown §1); src/gridflow/connectors/openmeteo/endpoints.py

---

### Live API probe at plan time to verify platform capability claims

Before writing any code, two `curl` calls to the ERA5 archive endpoint confirmed which hub-height wind variables are actually served. This settled a key design question (archive variable list) before it could become a hard-to-debug runtime null-fill issue.

**When to use:** Whenever a planned feature depends on a third-party API returning specific fields that are documented but whose availability is uncertain (API docs may be aspirational, or certain fields may only work on specific endpoint variants). Run a targeted curl probe and inspect the returned `units` field before finalising the variable list.
**Source:** F7.5-RESULTS.md (Pre-execution verifications V1); F7.5-VERIFICATION.md (Observable Truth #2)

---

### DERIVE_* class variables for optional silver derivations

`DERIVE_AIR_DENSITY: ClassVar[bool]` and `DERIVE_HDD_CDD: ClassVar[bool]` on `BaseOpenMeteoTransformer` signal which derived columns each transformer subclass should compute. The base class `_add_derived()` method reads these flags rather than branching on the role name.

**When to use:** When a shared base transformer has optional derived columns that apply to some but not all subclasses. Class variables are more declarative and testable than string comparisons in the base class. Each subclass is self-documenting about what it derives, and a contract test can assert the correct flag values.
**Source:** src/gridflow/silver/openmeteo/historical.py (DERIVE_AIR_DENSITY/DERIVE_HDD_CDD); F7.5-VERIFICATION.md (Observable Truth #6)

---

### Property tests for physical formula derivations

Parametrize over a T×P grid (6 temperatures × 3 pressures = 18 combinations), assert the output is within a physically plausible band, and assert the formula identity within floating-point tolerance (1e-9). A separate test asserts null propagation when an input is null.

**When to use:** Any silver column derived from a physical formula (ideal gas law, heat degree days, wind power law). Parametrized property tests catch boundary failures that example-based tests miss, and formula-identity assertions guard against operator-precedence bugs.
**Source:** tests/unit/test_openmeteo_air_density.py; F7.5-01-PLAN.md (Task breakdown §9)

---

### Dataset spec contract tests as structural guardrails

A dedicated test file (`test_openmeteo_dataset_specs.py`) asserts structural invariants on `DATASET_SPECS`: correct number of keys, location count per role, no `__` in location names, `extra_params` iff `global_tilted_irradiance` in hourly vars, archive and forecast wind vars differ and forecast is a superset.

**When to use:** After adding a spec table that encodes multiple invariants by convention (not enforced by the type system). Contract tests catch misconfiguration at test time rather than at runtime when an API request fails or data is silently malformed. These tests are especially valuable when the spec table is the source of truth consumed by several independent components.
**Source:** tests/unit/test_openmeteo_dataset_specs.py; F7.5-VERIFICATION.md (Threat #5, #6)

---

### Double-underscore as unambiguous compound key separator

When a per-instance identifier must embed both a multi-word dataset key and a location name (`{dataset}__{location.name}`), a double-underscore separator avoids the ambiguity of a single underscore where both components may contain single underscores.

**When to use:** Any compound key where either component is itself a multi-part underscore-delimited name. Pair with a contract test asserting that no individual component contains the separator character (preventing future collisions if a location name gains an underscore).
**Source:** F7.5-01-PLAN.md (Task breakdown — client.py); F7.5-VERIFICATION.md (Observable Truth #5, Threat #6)

## Surprises

### ERA5 archive silently returns all-null for 80/120/180m wind heights

The Open-Meteo ERA5 archive endpoint documents wind speeds at multiple hub heights but, in practice, returns `units: "undefined"` and all-null values for 80m, 120m, and 180m. Only 10m and 100m are reliably populated. This was not mentioned in the Open-Meteo docs or API changelog.

**Impact:** The entire `WIND_ARCHIVE_VARS` design had to restrict to 10m + 100m only. If this had been discovered at backfill time rather than during pre-execution verification, ~35,000 null-filled API calls would have been wasted and wind silver would have been silently useless above 10m. The live probe saved the equivalent of a full backfill wasted run.
**Source:** F7.5-RESULTS.md (Pre-execution verifications V1); F7.5-VERIFICATION.md (Observable Truth #3)

---

### F6 cloud_cover_pct field does not exist in this repo — F7.5-COMPAT-01 was moot

The proposed plan included a compatibility requirement `F7.5-COMPAT-01` referencing F6 and a `cloud_cover_pct` field. A `grep -r "cloud_cover_pct" .` returned zero matches — F6 is not present in this repo and `gridflow_models` appears only in `.planning/` documents.

**Impact:** The entire F6 compatibility surface was a non-issue. The requirement was rewritten as an internal-only test-pass requirement and `F6` dropped from `depends_on:`. This saved planning overhead but also exposed that the proposed plan was written from context that assumed F6 was in-repo when it wasn't.
**Source:** F7.5-RESULTS.md (Pre-execution verifications V2)

---

### Dead assertion in test silently passed for the wrong reason

`tests/unit/test_openmeteo_air_density.py:107` contained `assert "..." not in ... if False else True`. Python parses this as `assert (... if False else True)`, which always evaluates to `True`. The intended assertion was never executed. The test passed but was testing nothing. The actual check was inadvertently enforced by a later block in the same test function that used `__new__`-constructed transformer.

**Impact:** If the dead assertion had been the only check, the contract that solar transformers omit `air_density_kg_m3` would have had zero test coverage. Caught by code review (WR-02). Lesson: `if <const>` branches in assert statements are a red flag — `assert (expr if False else True)` is a footgun that no static linter catches by default.
**Source:** F7.5-REVIEW.md (WR-02)

---

### ServingClient and run_all_sources.py were invisible to the migration sweep

Both files reference dataset names by string literal, not by importing the module constants. A grep scoped to the connector/silver tree — the obvious migration target — missed them. The code reviewer found both within minutes by scanning `serving/` and `scripts/` with fresh eyes.

**Impact:** If the branch had been shipped without the review-fix commit, `GridflowClient.get_weather()` would have raised a DuckDB catalog error on every call, and the nightly local run script would have silently produced zero openmeteo bronze. Both are silent regressions that would have been hard to attribute to F7.5.
**Source:** F7.5-REVIEW.md (Summary + CR-01 + WR-01)

---

### Avast antivirus TLS interception breaks uv pip install but not Python httpx API calls

The local Avast antivirus intercepts TLS connections during `uv pip install`, causing certificate validation failures (`UnknownIssuer`). This meant `uv run pytest` could not be used after any `uv pip install` attempt. Workaround: run tests via `PYTHONPATH=src python -m pytest -x -q`. Live API calls through the connector (Python httpx) were not affected because Open-Meteo's certificate chain is one Avast already inspects without rejection; only `curl --ssl-no-revoke` is needed for direct verification calls.

**Impact:** An extra `PYTHONPATH=src` prefix on every test command throughout the phase. The workaround is documented in V1-CONTEXT.md and should be assumed for any future phase requiring `uv run` until Avast TLS inspection is reconfigured.
**Source:** F7.5-RESULTS.md (Avast workaround note); earlier V1-CONTEXT.md documentation
